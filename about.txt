1. PROJECT OVERVIEW
================================================================================

This is an AI-powered Security Operations Center (SOC) Analyst that autonomously
detects, classifies, and responds to security threats in real-time. It combines:

- Machine Learning (Isolation Forest) for anomaly detection
- Large Language Model (Groq Llama 3.3 70B) for intelligent reasoning
- LangGraph Agent Framework for orchestrated decision-making
- Learning System that improves from past decisions
- Multi-Tenant Architecture with organization-scoped data isolation
- JWT Authentication with role-based access control

TARGET USERS:
- Security Operations Centers (SOCs)
- DevSecOps teams
- Managed Security Service Providers (MSSPs)
- Organizations wanting to automate threat response

5. DATA FLOW
================================================================================

STEP 1: LOG GENERATION
  - Real application logs OR simulated via log_generator.py
  - Timestamps are real-time sequential (for proper graphing)
  - Written to /logs/app.log in JSON format

STEP 2: LOG INGESTION
  - Fluent Bit tails log file
  - Parses JSON, extracts fields
  - Forwards to Elasticsearch (soc-logs-YYYY.MM.DD)
  - OR: Direct API ingestion (POST /api/logs) to tenant indices

STEP 3: ANOMALY DETECTION
  Trigger methods:
  - Manual: POST /api/incidents/process
  - Automatic: POST /api/detection/start (every 30 sec)
  - On-demand: POST /api/anomaly/detect

  Process:
  1. Fetch logs from time window (default 5 min)
  2. Extract 9 features via FeatureEngineer
  3. Run Isolation Forest prediction
  4. Score normalized to 0-1, threshold check

STEP 4: AGENT WORKFLOW (if anomaly detected)

  4a. Log Analysis Agent:
      - Extract metrics: IPs, paths, status codes, auth failures
      - Call Groq LLM for threat indicator insights
      - Fallback to rule-based if LLM fails
      - Output: log_analysis with threat_indicators

  4b. Threat Classification Agent:
      - Input: log_analysis + anomaly features
      - Call Groq LLM for classification
      - Fallback to rule-based thresholds
      - Output: ThreatClassification (attack_type, severity, MITRE)

  4c. Decision Agent:
      - Input: classification + analysis + learning context
      - Call Groq LLM with full context
      - Determine recommended actions
      - Calculate confidence scores
      - Auto-approve logic:
        * Confidence >= 85% -> Auto-approve
        * Historical accuracy >= 90% AND confidence >= 70% -> Auto-approve
        * Safe actions + confidence >= 70% -> Auto-approve
      - Output: recommended_actions, requires_human_approval

  4d. Response Agent (if approved):
      - Execute selected action (simulated)
      - Record outcome in memory
      - Update incident status to EXECUTED

STEP 5: INCIDENT STORAGE
  - Complete incident indexed in tenant's ES index
  - Includes: anomaly, classification, actions, reasoning, agent_logs

STEP 6: DASHBOARD UPDATE
  - WebSocket broadcasts every 5 seconds
  - Real-time metrics and incident updates
  - Time-series graphs from sequential timestamps

6. AGENT SYSTEM
================================================================================

FRAMEWORK: LangGraph (from LangChain)

LLM: Groq Llama 3.3 70B Versatile
  - Temperature: 0.1-0.2 (deterministic)
  - Max tokens: 500-1000 per call
  - JSON response format enforced
  - Robust parsing with truncation handling

AGENT 1: LOG ANALYSIS
  File: agents/log_analysis_agent.py
  Purpose: Parse and analyze raw logs

  Metrics Extracted:
  - Total logs, unique IPs, unique paths
  - Top IPs by request count
  - Status code distribution
  - Auth failure count and ratio
  - Suspicious paths (/.git, /.env, /admin, SQL patterns)
  - Geographic distribution
  - Time distribution

  LLM Prompt: Asks for threat indicators and attack patterns
  Fallback: Rule-based threat indicator detection

AGENT 2: THREAT CLASSIFICATION
  File: agents/threat_classification_agent.py
  Purpose: Classify attack type and severity

  Attack Types:
  - BRUTE_FORCE (T1110): Password guessing
  - RECONNAISSANCE (T1595): Scanning/probing
  - DDOS (T1498): Denial of service
  - INJECTION (T1190): SQL injection, path traversal
  - XSS: Cross-site scripting
  - SUSPICIOUS_IP (T1090): Suspicious geolocations
  - AUTH_FAILURE (T1078): Authentication failures
  - ANOMALOUS_TRAFFIC (T1071): Unusual patterns

  Severity Levels:
  - LOW: Minor anomalies, first-time events
  - MEDIUM: Elevated concern, moderate metrics
  - HIGH: Confirmed attack patterns
  - CRITICAL: Active exploitation, extreme metrics

  Thresholds (rule-based fallback):
  - BRUTE_FORCE: 5-20 failures=LOW, 20-50=MEDIUM, 50-100=HIGH, >100=CRITICAL
  - DDOS: 30-100 req/s=MEDIUM, 100-200=HIGH, >200=CRITICAL
  - RECONNAISSANCE: 10-20 paths=MEDIUM, >20=HIGH

AGENT 3: DECISION
  File: agents/decision_agent.py
  Purpose: LLM-powered autonomous decisions

  Features:
  - Primary: Groq LLM reasoning with full context
  - Learning context from memory system
  - Historical similar incidents consideration
  - Confidence-based auto-approval

  Auto-Approval Criteria:
  1. Confidence >= 85%
  2. Recent accuracy >= 90% AND confidence >= 70%
  3. Safe actions (MONITOR, ALERT, RATE_LIMIT) AND confidence >= 70%
  4. Similar past incidents all successful
  5. Critical severity requires >90% confidence

  Fallback: Learned thresholds + rule-based action selection

AGENT 4: RESPONSE
  File: agents/response_agent.py
  Purpose: Execute defensive actions

  Actions:
  - BLOCK_IP: Block IP (default 24h, permanent for CRITICAL)
  - RATE_LIMIT: Throttle (default 30 req/min for 60 min)
  - ALERT: Send alert to SOC team
  - ESCALATE: Escalate to incident response
  - MONITOR: Enhanced monitoring (default 120 min)
  - NO_ACTION: No action needed

  Note: Actions are SIMULATED - no real network blocking

MEMORY SYSTEM:
  File: agents/memory.py
  Purpose: Learning and context

  Stores:
  - All decisions with features and confidence
  - Outcomes (success/failure) from feedback
  - Attack patterns grouped by type
  - IP reputation scores
  - Action effectiveness rates
  - False positive/negative tracking

7. MACHINE LEARNING
================================================================================

MODEL: Isolation Forest (Unsupervised Anomaly Detection)

Why Isolation Forest:
- No labeled data required (unsupervised)
- Works well with high-dimensional data
- Fast training and prediction
- Isolates anomalies rather than profiling normal

Parameters:
  n_estimators: 100
  contamination: 0.1 (10% expected anomalies)
  max_samples: 'auto'
  random_state: 42

FEATURES (9 dimensions):

1. ip_frequency
   - Requests per second from single IP
   - High = automated activity
   - Threshold: >5 req/sec anomalous

2. failed_login_ratio
   - Failed auth / total auth attempts
   - High = brute force
   - Threshold: >0.5 anomalous

3. time_deviation
   - Requests outside business hours (8-18)
   - Off-hours activity suspicious
   - Threshold: >0.7 anomalous

4. status_code_entropy
   - Shannon entropy of status codes
   - Low entropy = scanning
   - Threshold: <0.5 with error_rate >0.3

5. request_burst_rate
   - Max requests in 60-second window
   - DDoS indicator
   - Threshold: >10 anomalous, >200 critical

6. geo_anomaly_score
   - Suspicious countries (CN, RU, KP, IR)
   - Multiple countries = suspicious
   - Threshold: >0.5 anomalous

7. unique_paths_ratio
   - Unique paths / total requests
   - Reconnaissance indicator
   - Threshold: >0.8 anomalous

8. avg_latency
   - Average response time (ms)
   - Service degradation
   - Threshold: >1000ms anomalous

9. error_rate
   - Proportion of 4xx/5xx responses
   - Attack indicator
   - Threshold: >0.5 anomalous

Training Process:
  1. Fetch historical logs (configurable hours)
  2. Split into windows (50 logs each)
  3. Extract 9 features per window
  4. Fit StandardScaler
  5. Fit IsolationForest
  6. Save to data/anomaly_model.pkl

8. MULTI-TENANCY & AUTHENTICATION
================================================================================

AUTHENTICATION FLOW:

1. Registration (POST /api/auth/register):
   - Create user account
   - Check for pending invitation
   - If no invitation: auto-create personal organization
   - If invitation exists: skip org creation, user can accept invite

2. Login (POST /api/auth/login):
   - Verify credentials
   - Return JWT token with org_id, org_role
   - Token valid for 24 hours

3. Invitation Flow:
   - Admin/Owner invites via email
   - Invitation stored with token (expires 7 days)
   - Invitee registers -> no auto-org creation
   - Invitee accepts -> joins organization with assigned role

ORGANIZATION ROLES:

| Role    | Permissions                                    |
|---------|------------------------------------------------|
| OWNER   | Full access, manage org, delete org, transfer  |
| ADMIN   | Manage members, approve all incidents          |
| ANALYST | View incidents, approve assigned incidents     |
| VIEWER  | Read-only access to dashboard                  |

DATA ISOLATION:
- Each organization has separate ES indices
- Queries scoped by org_id from JWT
- WebSocket updates filtered by organization
- No cross-organization data access

12. CURRENT LIMITATIONS
================================================================================

A) SIMULATED ACTIONS
   - IP blocking is mock (not actually blocked)
   - Rate limiting is mock (not enforced)
   - No real firewall/WAF integration
   - Actions stored in memory only

B) PERSISTENCE
   - Learning memory is in-process (lost on restart)
   - Blocked IPs lost on restart
   - No Redis/database backing for memory

C) SCALABILITY
   - Single-node deployment only
   - No horizontal scaling
   - No message queue for high throughput

D) ALERTING
   - No real alert delivery (Slack, email, PagerDuty)
   - Alerts stored in memory only

E) LLM DEPENDENCIES
   - Requires Groq API key
   - Rate limits on Groq API
   - No local LLM option

F) TESTING
   - Limited unit tests
   - No integration tests
   - Relies on simulated logs
